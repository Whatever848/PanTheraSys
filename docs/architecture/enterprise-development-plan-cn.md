# PanTheraSys 企业级开发计划

## 1. 当前结论

`PanTheraSys` 目前已经不是空项目，而是一个具备企业级雏形的 `Qt 6.2.0 + C++ + CMake + MySQL 5.7` 原型仓库。现阶段已经具备：

- 分层目录结构：`apps / src/core / src/adapters / src/modules / tests / docs`
- 桌面工作站壳层：设备监控、治疗方案、治疗执行、数据管理 4 个主工作区
- 安全内核：患者未选择、方案未审批、水循环/功率/运动/急停异常等联锁判定
- 仿真设备后端：实时状态快照、故障注入、治疗输出模拟
- 基础数据模型：患者、影像、治疗方案、治疗记录、审计、报警等
- MySQL 5.7 初始化脚本
- GitHub 工程底座：CI、CodeQL、PR/Issue 模板、UTF-8 校验脚本
- QMS 起步文档：需求基线、软件设计说明、风险登记、追溯矩阵、验证计划

## 2. 安全边界

当前仓库仍然属于“工程原型 / 演示原型”，适合：

- 产品方案评审
- UI 走查
- 软硬件联调前的流程验证
- 模块拆分和企业级工程化建设

当前仓库不适合：

- 用于人体治疗
- 作为注册完成的软件直接投入临床
- 在未完成 V&V、风险管理闭环、硬件联调、EMC/电气安全验证前用于真实设备控制

## 3. 结合需求文档后的模块拆分

根据 `开发功能模块分化.docx` 和 `应化所乳腺癌系统需求确认文档.xlsx`，完整系统建议拆为 6 大业务域：

1. 数据管理域
   患者信息、影像数据、治疗数据、治疗报告的 CRUD、查询、归档与审计
2. 图像采集与处理域
   DICOM 接入、图像采集参数配置、图像本地存储/读取、OpenGL 显示、后续图像算法接口
3. 运动控制域
   9 轴控制、位置校准、参数存储、手柄/计算机双模式控制、线程隔离
4. 肿瘤定位与三维重构域
   肿瘤边界标定、三维重建、病灶轮廓与治疗范围评估
5. 治疗方案与治疗执行域
   点治疗、线治疗、分段执行、呼吸跟随、治疗暂停/恢复/终止、进度反馈
6. 设备监控与安全域
   电源、水循环、换能器、运动系统、能量发生器、图像质量监控与联锁报警

## 4. 当前差距分析

### 已完成或部分完成

- 主工作站导航与主题风格
- 方案页、治疗页、设备监控页原型
- 安全联锁核心逻辑
- 仿真设备适配器
- MySQL 表结构基础设计
- 审计服务与基础测试

### 关键缺口

- 尚未接入真实 MySQL CRUD 仓储层
- 尚未接入 DICOM/DCMTK、OpenGL 专用图像视图、OpenCV 算法链路
- 尚未接入真实 9 轴运动控制器与设备通信协议
- 尚未实现治疗评估、三维重建、病灶边界编辑
- 尚未实现用户登录、权限持久化、签名审批、电子留痕
- 尚未建立系统化的集成测试、HIL 测试、异常恢复测试
- 尚未形成注册导向的软件配置项管理和完整追溯体系

## 5. 推荐目标架构

建议在现有分层结构上继续演进为以下形态：

- `UI Shell`
  Qt Widgets 工作站壳层、导航、页面装配、角色可见性控制
- `Application`
  患者上下文、方案上下文、治疗会话、用例编排、事件总线
- `Domain`
  患者/影像/治疗/设备/安全联锁/审计等领域模型和规则
- `Infrastructure Adapters`
  MySQL、DICOM、OpenGL、OpenCV、运动控制器、功率控制器、水循环监控、配置管理
- `Quality & QMS`
  风险、追溯、验证、发布、变更记录、GitHub CI/CD

## 6. 分阶段开发路线

### M1 已完成

- 工程底座
- 原型页面
- 安全内核
- 仿真后端
- GitHub 基础协作框架

### M2 数据域固化

- 建立临床数据仓储接口
- 完成患者档案、影像、治疗、报告的数据实体与页面联动
- 接入 MySQL Repository 与基础查询/新增/修改/删除
- 完成数据管理页面与审计闭环

### M3 图像采集与处理

- 引入 DICOM SCP 接收服务
- 集成 DCMTK
- 封装统一 `OpenGLView` 父类
- 完成图像读取、显示、缓存、本地归档
- 为后续肿瘤边界标注与呼吸跟随提供接口

### M4 设备监控与运动控制

- 定义真实硬件适配器接口
- 接入 9 轴控制、原点回归、校准、位置补偿
- 完成功率、水循环、换能器、电机状态的实时线程模型
- 建立故障码、报警、恢复流程

### M5 治疗方案与评估

- 线治疗/点治疗/分段治疗路径生成
- 治疗参数保存与版本管理
- 肿瘤边界标定、三维重建、方案评估
- 呼吸跟随算法框架

### M6 治疗执行闭环

- 治疗开始/暂停/恢复/终止
- 治疗进度、能量、时间、剂量实时记录
- 治疗记录与报告自动生成
- 方案审批、锁定、执行回放

### M7 工程化加固与注册准备

- 用户认证、权限、审批签名、配置审批
- 单元测试、集成测试、系统测试、异常测试、HIL 测试
- 风险控制闭环与追溯矩阵补全
- GitHub release、版本标签、发布包、部署文档

## 7. GitHub 仓库交付要求

仓库已经有基础骨架，建议继续按以下规范推进：

- 保护 `main`
- 日常开发使用 `feature/*`
- 每个功能分支必须通过 PR 合并
- 启用必须检查：Windows 构建、CTest、UTF-8 校验、CodeQL
- 图像和大文件使用 Git LFS
- 运行数据、患者敏感数据、设备日志不得直接提交仓库
- 每个 PR 必须绑定需求编号、风险编号、测试说明

## 8. 近期两轮冲刺建议

### Sprint 1

- 打通数据管理页与临床种子数据
- 建立患者切换时的方案安全失效机制
- 增加数据仓储接口，去除页面硬编码
- 补关键单元测试

### Sprint 2

- 接入 MySQL 持久化仓储
- 完成患者/影像/治疗/报告四类数据 CRUD
- 建立配置文件与数据库连接的环境切换
- 初步形成数据管理模块的企业级闭环

## 9. 后续强制要求

项目如果目标是最终用于人体治疗，后续开发必须以“注册与验证导向”推进，而不是单纯把界面做完整。所有后续模块都应补齐：

- 需求编号
- 风险控制点
- 设计说明
- 测试项
- 审计留痕
- 版本记录

只有这样，这个仓库才能从“好看的原型”真正变成“可交付的医疗设备软件工程项目”。
